# 多独立域名 + Cloudflare 共用同一套账号

本文与 [`DEPLOY-CLOUDFLARE.md`](DEPLOY-CLOUDFLARE.md)、[`账号体系需求规格-复刻用.md`](账号体系需求规格-复刻用.md)、[`验证与接入指南.md`](验证与接入指南.md) 配套，约定**已选定**的实施方案与配置项；**是否要测、新站怎么接**见验证与接入指南。

## 选定方案

| 项 | 选择 |
|----|------|
| 主方案 | **方案 A**：单一 API 入口（本仓库 Pages + `functions/`）+ 各业务站通过 HTTPS 跨域调用 |
| 备选 | **方案 B**：各站各自部署相同 Functions，共用同一 D1 `database_id` 与同一 `JWT_SECRET`（需保持代码同步） |
| 单点登录（SSO） | **未在本仓库实现**；若需「只登一次、多站通用」，需在集中登录域上增加回调换 token（短期 code 优于 URL 明文 token）。见下文「可选：集中登录扩展点」 |

## Auth API 与前端根地址

- **API 基址（示例）**：`https://qbt-datavisualization.pages.dev`（以实际部署域名为准）。
- **业务站**：独立域名，例如 `https://other-app.com`。
- 业务站前端应使用与 [`js/auth-client.js`](js/auth-client.js) 相同约定：
  - 生产环境若页面与 API **不同源**：在构建或运行时把 API 根设为上述基址；本地/预览可用 `localStorage.setItem('QBT_API_ORIGIN', 'https://你的API域名')` 覆盖 [`REMOTE_API_ORIGIN`](js/auth-client.js) 的默认值。

## CORS 白名单（必选，多域跨站时）

在 **承载 `/api/*` 的** Cloudflare Pages 项目中配置：

```text
ALLOWED_ORIGINS=https://qbt-datavisualization.pages.dev,https://业务站1.example,https://业务站2.example
```

规则见 [`functions/_lib/http.js`](functions/_lib/http.js) 内 `resolveCorsOrigin`：仅当请求的 `Origin` 在白名单中时回显 `Access-Control-Allow-Origin`。**不要**在带鉴权头的场景误用 `*`。

## D1 与 JWT（同一套账号数据）

- **一个** D1 数据库保存 `users` 等表；`JWT_SECRET` 仅在与该库配套的 API 项目上配置（方案 A）。
- 若采用方案 B：多个项目绑定**同一** `database_id`，且各项目 `JWT_SECRET` **完全一致**。

## 可选：集中登录扩展点（SSO，未实现）

若产品要求跨顶级域共享登录态，典型做法是：

1. 在 `auth.example.com` 完成登录并签发 JWT（仍用同一 D1 + `JWT_SECRET`）。
2. 重定向回 `https://业务站/auth/callback`，用 **短期一次性 `code`** 换 token，由业务站前端写入**本站** `localStorage`（与现有 `xbs_token` 键一致即可）。

本仓库未提供 `code` 表与换发路由；新增时需单独设计 `oauth_codes` 表 TTL 与一次性消费逻辑。

## 验收

- 业务站浏览器请求 `POST /api/auth/login` 等接口时，响应头包含与请求 `Origin` 一致的 `Access-Control-Allow-Origin`（且该 Origin 已列入 `ALLOWED_ORIGINS`）。
- 未列入白名单的 Origin 在已配置 `ALLOWED_ORIGINS` 时无法完成跨域读 JSON。
