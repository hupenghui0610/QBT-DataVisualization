## 迁移到其他技术栈时的映射（非 Cloudflare 时替换名称）

| 本规格 / 本仓库 | 迁移时等价概念 |
|------------------|----------------|
| Cloudflare Pages Functions | 任意 HTTPS API（Node、Bun、Go 等） |
| D1（SQLite） | 任意关系库；表结构可沿用 SQLite DDL 或迁移为 PostgreSQL/MySQL 类型 |
| 绑定名 `env.DB` | 你的 ORM / 数据库连接 |
| `request.cf`（城市/国家） | 可选：GeoIP 库或反向代理注入的请求头 |
| `CF-Connecting-IP` / `X-Forwarded-For` | 统一从可信代理取客户端 IP |
| `JWT_SECRET` | 与 HS256 实现一致的环境变量密钥 |
| `localStorage` + `xbs_token` | 可改为 `sessionStorage` 或 httpOnly Cookie（需同步调整 XSS/CSRF 策略） |

行为与数据结构保持不变；仅运行时与存储驱动按目标栈重写。

## 1. 产品规则（账号生命周期）

- **无自助注册**：登录页文案说明「暂未开放注册」，账号由管理员通过接口创建。
- **账号标识**：业务上「账号」= **11 位中国大陆手机号**（`/^1\d{10}$/`），全局唯一。
- **用户展示名**：`name`（姓名），登录后顶栏显示优先 `name`，否则回退 `phone`。
- **角色**：二元标记 `is_admin`（管理员 / 普通用户）。**创建用户接口固定创建为普通用户**（`is_admin = 0`）；管理员账号需数据库或迁移脚本写入 `is_admin = 1`。
- **密码策略**：新密码 / 初始密码 **至少 6 位**；服务端与前端均校验。

## 2. 数据存储（D1 / SQLite）

**表 `users`**（见 [migrations/0001_init.sql](migrations/0001_init.sql)）

| 字段 | 说明 |
|------|------|
| `id` | 自增主键，JWT `sub` |
| `name` | 姓名 |
| `phone` | 手机号，UNIQUE |
| `password_hash` | 见下文哈希格式 |
| `token_version` | 整数，用于 **作废全部已签发 JWT**（改密时 +1） |
| `is_admin` | 0/1 |
| `created_at` | 创建时间 |

**表 `access_logs`**：访问审计（见 ping 逻辑）。

**表 `login_security_events` / `login_security_blocks`**：登录安全事件与临时封禁（见 [migrations/0002_login_security.sql](migrations/0002_login_security.sql)）。

---

## 3. 密码哈希（与 Node 脚本一致）

- 算法：**PBKDF2-HMAC-SHA256**，迭代 **100000**，派生 **32 字节**。
- 存储字符串格式：`pbkdf2$sha256$100000$<salt_hex>$<hash_hex>`（见 `hashPassword`）。
- 校验：常量时间比较派生字节与存储哈希（见 `verifyPassword`）。
- 运维：手工建用户时用仓库 [scripts/hash-password.mjs](scripts/hash-password.mjs) 生成哈希再 `INSERT`（勿把明文密码提交到仓库）。

---

## 4. JWT 与会话

- 算法：**HS256**，密钥来自环境变量 **`JWT_SECRET`**（未配置时登录返回 500）。
- **有效期**：`JWT_EXP_SECONDS = 30 * 24 * 60 * 60`（**约 30 天**，不是 24 小时；若你方文档写 24h 需与实现统一）。
- **Payload 字段**：`sub`(用户 id)、`phone`、`name`、`adm`(管理员为 1)、`tv`(**必须与 DB `token_version` 一致**)、`iat`、`exp`。
- **服务端鉴权**（`authenticateRequest`）：解析 Bearer → 验签 → 按 `sub` 读库 → **比对 `payload.tv` 与 `token_version`**，不一致则 401「登录已失效」。
- **改密副作用**：`UPDATE users SET password_hash = ?, token_version = token_version + 1`，使旧 JWT 全部失效。

---

## 5. 登录流程与安全策略

**接口**：`POST /api/auth/login`，Body：`{ "phone", "password" }`。

1. 检查 `JWT_SECRET`。
2. **封禁预检**（`getCurrentLoginRestrictions`）：若 IP 或手机号在 `login_security_blocks` 且 `blocked_until` 未过期 → 429，并写 `login_rejected` 事件。
3. 查用户；不存在或密码错误均返回 **401「账号或密码错误」**（不区分），并走失败计数。
4. **失败处理**（`registerFailedLogin`）：记录 `login_failed`（account + ip 两条）；按窗口计数可能 **锁定账号** 或 **封禁 IP**，并写 `account_locked` / `ip_blocked` 等事件。
5. **成功**（`registerSuccessfulLogin`）：清除该账号的 block、写 `login_success`。
6. **响应延迟**（`failedLoginDelayMs`）：失败次数较高时 sleep 1s/2s/4s，缓解撞库。
7. **关键阈值常量**（可按需在新项目调整但需成文档）：短窗口 10 分钟内账号失败 ≥3 → 锁 5 分钟；长窗口 20 分钟内 ≥5 → 锁 15 分钟；IP 10 分钟内失败 ≥20 → 封 30 分钟（见 `login-security.js` 顶部常量）。

## 6. 其他认证相关 API（行为要点）

| 路径 | 方法 | 作用 |
|------|------|------|
| `/api/auth/change-password` | POST | Bearer；`oldPassword` / `newPassword`；成功则 `token_version++`，返回 `{ ok: true, message: '密码已更新，请重新登录' }` |
| `/api/auth/ping` | **POST**（客户端如此调用） | Bearer；返回 `{ ok: true, user }`；**附带访问日志逻辑**（见下） |

**与 [后台接口描述.md](后台接口描述.md) 的差异**：文档写 ping 为 GET、响应为 `valid` 等——**以代码为准**（POST + `ok`）。

**Ping 与访问日志**（[functions/api/auth/ping.js](functions/api/auth/ping.js)）：

- 从 `request.cf` 取 `city`/`country`，`CF-Connecting-IP` / `X-Forwarded-For` 为 IP。
- **特例**：若用户 `name` 经 trim 后等于 **`胡鹏辉`**，**不写** `access_logs`。
- 否则：若 **过去 4 小时内** 不存在「同一 `user_id` + 同一 `client_ip`」的记录，则插入一条（`accessed_at` 为 UTC 分钟对齐），用于去重、减轻刷屏。

**管理员接口**（均需登录且 `is_admin`）：

- `GET /api/admin/access-logs?limit&offset`：分页访问记录（join users）。
- `GET /api/admin/login-security-events?limit&offset`：分页安全事件（**实现未按文档支持 scope_type/event_type 筛选**，返回全表倒序分页）。
- `GET /api/admin/users`：用户列表。
- `POST /api/admin/users`：`name, phone, password`；手机号校验、查重；**新用户恒为普通用户**。

## 7. 前端：存储与跨域

**客户端模块** `XbsAuth`（[js/auth-client.js](js/auth-client.js)）：

- Token 存 **`localStorage` 键 `xbs_token`**。
- 需鉴权请求：`Authorization: Bearer <token>`，`Content-Type: application/json`。
- **本地/局域网/file 协议打开页面**：API 基址指向固定生产源 `https://qbt-datavisualization.pages.dev`；可通过 `localStorage['QBT_API_ORIGIN']` 覆盖为其他 **https** 根（用于预览环境）。
- 提供：`login`、`ping`、`changePassword`、`clearSession`、以及管理员与数据拉取等封装。

---

## 8. 前端：界面与交互（主站 index）

**登录门 (`#auth-gate`)**  
全屏遮罩 + 卡片：标题「希倍思内部平台」、提示联系开通、手机号 + 密码、`进入` 按钮、错误文案区。

**启动逻辑 (`bootstrapAuth`)**  

- 无 `XbsAuth` → 只显示登录门。
- 无 token → 显示登录门并绑定表单。
- 有 token → `POST ping`：成功则 `currentUser = data.user`，`updateUserBar()`，`enterApp()`；失败或网络错误 → `clearSession()` 并回登录门。

**顶栏 (`#userSessionBar`)**  
显示姓名；按钮：**修改密码**、**退出登录**；管理员额外：**账号管理**、**访问记录**、**登录安全**。

**修改密码模态框**  
原密码、新密码、确认；前端校验长度与两次一致；成功后 **清 token、清 currentUser、关模态、回登录门**（与后端 `token_version` 递增一致）。

**管理员模态框**  

- **访问记录**：表格姓名/手机/时间/IP，分页。
- **登录安全事件**：时间、事件类型、范围、账号、IP、附加信息（`meta_json` 解析展示），分页。
- **账号管理**：用户列表（姓名、手机、身份）；表单新增用户（姓名、手机、初始密码）。

**401 处理**：业务接口若返回 401，前端应在数据加载层统一处理（本项目多处 fetch 需自行对齐；复刻时建议集中封装）。

## 9. 权限模型（后端）

- **默认**：所有业务数据接口经 `authenticateRequest` 要求登录。
- **管理员**：`isAdmin(user)` 为真时允许 `/api/admin/*` 等；非管理员 403「无权访问/操作」。

## 10. 部署与环境

- **运行时**：Cloudflare Pages Functions（Edge）+ **D1 绑定名 `DB`**。
- **必填密钥**：`JWT_SECRET`。
- **IP 识别**：依赖 `CF-Connecting-IP` / `X-Forwarded-For`（本地 wrangler 行为可能与生产不一致，复刻时需约定）。
- **CORS**：鉴权错误也需带 CORS 头，否则浏览器跨域场景会表现为 `Failed to fetch`（见 [DEPLOY-CLOUDFLARE.md](DEPLOY-CLOUDFLARE.md) 说明）。

### 10.1 多独立域名场景下的非功能需求（与方案 A/B 对齐）

复刻或多站接入时，须在需求/运维文档中**显式写清**下列两项（本仓库已实现对应机制，见 [多域名共用账号-部署说明.md](多域名共用账号-部署说明.md)）：

1. **API 根域名策略**  
   - **方案 A（推荐）**：仅一个 Pages/Worker 部署承载 `/api/*` 与用户库；各业务站前端通过 [`js/auth-client.js`](js/auth-client.js) 将请求发到该入口——生产可改 **`REMOTE_API_ORIGIN` 默认值**，或通过 **`localStorage['QBT_API_ORIGIN']`**（键名 `QBT_API_ORIGIN`）按环境覆盖。  
   - **方案 B**：各站各自部署同源 `/api/*`，但共用**同一 D1 `database_id`** 与**同一 `JWT_SECRET`**；此时各站 `getApiBase()` 可为空（与页面同源）。  
   - 无论 A/B，均须在文档中列出：**正式 / 测试 / 本地** 各环境最终使用的 API 根 URL。

2. **CORS 白名单（浏览器跨域时）**  
   - 在承载 API 的项目上配置 **`ALLOWED_ORIGINS`**：逗号分隔的**完整 Origin**（含协议与端口），列出**所有**允许调用该 API 的独立业务域名（例如 `https://a.com,https://b.com`）。  
   - 实现见 [`functions/_lib/http.js`](functions/_lib/http.js) 的 `resolveCorsOrigin`：**未配置**时与旧行为兼容；**已配置**时仅白名单内 Origin 会收到 `Access-Control-Allow-Origin`。

## 12. 复刻验收清单（给实现方自测）

- 登录成功返回 token + `user`（含 `id, name, phone, is_admin`）。
- 改密后旧 token 立即失效；前端回到登录。
- 失败登录触发延迟与可能的 IP/账号封禁；管理员可查阅安全事件表。
- Ping 在非特例用户下按规则写入/去重访问日志；管理员可查阅。
- 本地打开页面时 API 指向可配置远程源、Token 持久化在 localStorage。
