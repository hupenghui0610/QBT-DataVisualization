# 电商数据可视化平台搭建经验总结

---

## 一、项目概述

### 1.1 这是一个什么样的项目

本项目是一个**电商数据可视化大屏系统**，可以把散落在飞书表格里的销售数据，自动转换成直观的图表和趋势分析。

**通俗理解**：想象你有很多张Excel表格记录着每天的销售额，本项目就像是一个"自动报表生成器"，能把这些数字变成漂亮的折线图、柱状图，让你在手机上或电脑上随时查看。

### 1.2 项目解决的核心问题

- **数据分散**：销售数据存在飞书表格里，查看不方便
- **人工统计**：每周每月汇总数据需要手动计算，容易出错
- **缺乏可视化**：纯数字表格难以发现趋势和问题

## 三、技术栈与服务简介

### 3.1 前端展示层（用户看到的内容）

- **HTML**：网页的骨架，定义页面结构
- **JavaScript**：网页的大脑，处理交互逻辑

**学习难度**：★★☆☆☆

### 3.2 后端服务层（数据处理）

- **是什么**：免费的网站托管服务
- **作用**：让你的网站可以被全世界访问

- **是什么**：Cloudflare提供的轻量级数据库
- **作用**：存储用户账号、登录信息等
- **特点**：免费、无需维护服务器

## 四、数据流转全流程

### 步骤一：数据准备（飞书表格）

**做什么**：在飞书里创建表格，按规范录入数据

**关键点**：
- 表格需要有规范的列名（如日期、销售额、渠道等）
- 数据格式要统一（如日期都用YYYY-MM-DD格式）
- 给程序创建一个专门的"应用账号"，让它能访问表格

**耗时**：取决于数据量，通常1-2小时

### 步骤二：编写"云端秘书"（Cloudflare Functions）

**做什么**：写代码告诉系统怎么从飞书取数据

**核心逻辑**：
1. 接收前端请求（如"我要看2026年1月的GMV数据"）
2. 拿着飞书密钥，去飞书API取原始数据
3. 对数据进行计算（如按渠道汇总、计算退款率等）
4. 把处理好的数据返回给前端

**代码示例（简化版）**：
```javascript
// 当用户请求GMV数据时
export async function onRequestGet(context) {
  // 1. 验证用户是否登录
  const auth = await authenticateRequest(request);
  
  // 2. 从飞书获取原始订单数据
  const rawData = await fetchSheetValues(env, spreadsheetToken, range);
  
  // 3. 按日期和渠道汇总计算GMV
  const result = aggregateByDayAndCategory(rawData);
  
  // 4. 返回给前端
  return jsonResponse({ gmv: result });
}
```

**耗时**：初次编写2-3天，后续优化1-2小时

### 步骤三：制作可视化页面（前端开发）

**做什么**：编写网页代码，把数据变成图表

**核心逻辑**：
1. 用户打开网页，输入账号密码登录
2. 页面自动向"云端秘书"请求数据
3. 拿到数据后，用ECharts生成图表
4. 用户可以通过按钮切换日度/月度、选择日期范围

**关键代码片段（简化版）**：
```javascript
// 从后端获取数据
const response = await fetch('/api/data/feishu-newretail-daily');
const data = await response.json();

// 用ECharts生成折线图
const chart = echarts.init(document.getElementById('chart'));
chart.setOption({
  xAxis: { data: data.dates },  // X轴：日期
  series: [
    { name: '京东', data: data.jd, type: 'line' },
    { name: '天猫', data: data.tmall, type: 'line' },
    { name: '抖音', data: data.douyin, type: 'line' }
  ]
});
```

**耗时**：制作一个图表约2-4小时

### 步骤四：部署上线

**做什么**：把代码发布到Cloudflare，让所有人都能访问

**操作步骤**：
1. 在Cloudflare注册账号
2. 创建Pages项目
3. 绑定GitHub仓库（代码会自动同步）
4. 配置环境变量（飞书密钥等敏感信息）
5. 点击"部署"

**耗时**：首次配置30分钟，后续更新只需1分钟

## 五、核心功能模块详解

### 5.1 全渠道GMV/GSV趋势图

**功能描述**：
- 显示京东、天猫、抖音等多个渠道的销售趋势
- 支持日度/月度切换
- 支持选择时间范围

**技术要点**：
- 不同渠道的数据单位要统一（元→万元）
- 新零售数据（来自飞书）和货架电商数据（来自京东/天猫后台）单位不同，需要分别处理
- 月度模式下要对当月数据做汇总

### 5.2 新零售四平台图表

**功能描述**：
- 把抖音、小红书、视频号、快手四个平台的数据合并展示
- 按DP、直对、服务商三个维度拆分

**技术要点**：
- 四个平台的数据格式可能不同，需要统一处理
- 渠道映射表需要维护（达人ID对应哪个渠道）
- GMV和GSV的计算口径不同（GSV要剔除取消订单）

### 5.3 月度累计达成

**功能描述**：
- 显示当月各渠道的GMV、GSV对比
- 计算整体退款率

**技术要点**：
- 需要读取整月数据进行累加
- 退款率 = (1 - GSV/GMV) × 100%

## 六、调试经验与常见问题

### 6.1 数据不显示/为空

**排查步骤**：
1. 打开浏览器开发者工具（F12），看Console有没有报错
2. 检查网络请求（Network标签），看API返回了什么
3. 确认飞书表格格式是否正确
4. 检查环境变量是否配置正确

**常见原因**：
- 飞书表格列名不对
- 数据格式错误（如日期格式不统一）
- 环境变量没配或配错了

### 6.2 图表显示不正确

**排查步骤**：
1. 检查数据单位是否统一（元 vs 万元）
2. 确认数据类型（字符串 vs 数字）
3. 查看ECharts配置是否正确

**常见原因**：
- 单位不统一导致数据差异过大（如京东是元，新零售是万元）
- 数据以字符串形式存储，需要转换为数字

### 6.3 部署失败

**常见错误**：
- "Unable to resolve Cloudflare's API hostname"：网络问题，稍后重试
- "Authentication error"：登录状态过期，重新运行 `npx wrangler login`

### 6.4 调试技巧

**前端调试**：
- 用 `console.log()` 打印数据，看实际拿到了什么
- 用断点调试，逐步执行看变量值

**后端调试**：
- 查看Cloudflare Functions的日志
- 在关键位置加日志输出

## 十、总结与心得

### 10.1 项目核心价值

- **自动化**：告别手动复制粘贴数据
- **实时性**：数据更新后刷新页面即可看到最新结果
- **可视化**：一眼看出趋势和问题

### 10.2 技术选型建议

- **小团队/个人**：推荐本方案（Cloudflare Pages + Functions），完全免费
- **大团队**：可以考虑专业BI工具（如Tableau、PowerBI）

### 10.3 最重要的经验

1. **数据规范是关键**：飞书表格的格式一定要规范统一，否则后续处理很麻烦
2. **先跑通再优化**：先让基本功能跑起来，再考虑优化性能
3. **善用日志**：前后端都要加日志，出问题才好排查
