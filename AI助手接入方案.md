# 报表 AI 助手接入方案（国内模型版）

## 目标与约束

- **目标**：在 index.html 及 charts.html 中增加 AI 助手，用户（中文）提问后，**每次回答都基于当前页面已加载的两份数据**（大盘、分品牌）生成；全流程为中文交互。
- **回答范围**：助手**只针对两份数据本身**做解读与回答，不涉及页面布局、界面设计、以及页面上已有的图表/报表形态；用户问的是「数据问题」就按数据答，不介绍或解释本页有哪些图、怎么用筛选等。
- **约束**：站点为静态页，部署在 Cloudflare Pages；无现有后端。数据在浏览器内存中（`dataSource` / `getDaPan()`、`getFenPinPai()`），体量大，不能全文塞给模型，需做**摘要再注入**。部署与用户均在国内，需选用**国内可访问、中文能力强的模型**。

---

## 1. 国内模型选择推荐

以**中文效果与国内可用性**为主，推荐以下国内大模型（任选其一，后端通过环境变量配置即可切换）：

| 厂商 | 典型产品 | API 特点 | 备注 |
|------|----------|----------|------|
| **阿里云 通义千问** | qwen-turbo / qwen-plus | DashScope 开放平台，按 token 计费 | 文档完善，国内延迟低 |
| **百度 文心一言** | ernie-bot / ernie-bot-turbo | 千帆大模型平台 API | 中文场景成熟 |
| **智谱 AI** | chatglm-turbo / glm-4 | 开放平台，兼容 OpenAI 格式 | 接口简单，易对接 |
| **月之暗面（Moonshot）** | moonshot-v1 | 开放 API，长上下文 | 适合带较长数据上下文 |
| **DeepSeek** | deepseek-chat | 开放 API，性价比高 | 中文好、价格低 |
| **腾讯混元** | hunyuan-standard | 云 API | 需腾讯云账号 |

**推荐优先**：**通义千问**（阿里 DashScope）或 **智谱 ChatGLM**——文档清晰、国内直连、中文表现稳定。若希望**长上下文**（数据摘要较长）可考虑 **月之暗面**。

## 2. 整体架构

- **前端**：默认只显示一个**悬浮按钮**（如右下角「AI 助手」图标），点击后展开**悬浮的聊天框**；聊天框内为对话区与输入框。每次发送前用当前内存中的 `dataSource` 生成**数据上下文字符串**并随问题一起发给后端。界面与提示语均为中文。
- **后端**：部署一个**代理服务**，接收 `question` + `context`，将 context 写进 system prompt，调用**国内大模型 API**（通义/智谱/文心等），把回复返回前端。API Key 仅存于后端，前端不接触。

这样**每次提问都会带上当前两份数据的摘要**，模型只根据这段上下文用中文回答，避免幻觉；若用户切换或上传了新数据，下次提问会自动用新数据。

**后端方案（已选）**：采用 **Cloudflare Worker** 作为后端代理。与报表页同属 Cloudflare，后续任何后端修改（如改 system prompt、换模型、修逻辑）只需更新 Worker 代码并重新部署（`wrangler deploy`）即可，无需维护国内云函数或其它服务器。Worker 内通过 `fetch` 调用国内大模型 API（通义/智谱等），API Key 放在 Worker Secret 中；需确认所选国内 API 允许境外 IP 调用，国内用户访问 Worker 时可能有轻微延迟。

## 3. 如何实现「每次提问都基于两份数据回答」

### 3.1 数据上下文从哪里来

- 页面内已有 `getDaPan()`、`getFenPinPai()`，对应**当前已加载的大盘与分品牌数据**（含用户上传 Excel 后的结果）。
- 在发送请求前，由前端调用一个**构建函数**，把这两份数据转成一段**纯文本摘要**（不要直接塞几万行 JSON），作为本次请求的 `context`。

### 3.2 上下文应包含什么（完整覆盖、紧凑格式）

**数据体量说明**：分品牌 JSON 约 57 万字符、大盘 default-data 约 93 万字符，原始全量远超常见模型上下文（如 32k tokens 约 10 万字符级）。因此「完整获取」采用**完整覆盖 + 紧凑格式**：不省略任何品牌、任何维度、任何关键指标，用表格/列表形式压缩呈现，使模型在单次请求内能看到**全部品牌与全部维度的数据**。

**大盘侧**（来自 `getDaPan()`）：

- `数据说明`：数据源、原始行数、时间范围等。
- 各维度**全部保留**，用紧凑格式：维度一各渠道一行（渠道、销量、销售额、占比、客单价）；维度二各价格段一行；维度三按日期可保留全部月份或最近 N 个月（视总长而定），每行「日期、销量、同比」；维度四～十同理，**不删维度、不删渠道/价格段**，仅将 JSON 压成易读的表格或列表。

**分品牌侧**（来自 `getFenPinPai()`）：

- 数据说明、前十大品牌名单（**必须完整**）。
- **品牌销量/销额**：10 个品牌每品牌一行（品牌、销量、销售额、销量占比、销售额占比），不省略。
- **品牌渠道占比**：每个品牌、每个渠道的占比均保留，用「品牌 \t 天猫 \t 京东 \t 抖音」等表格形式。
- **品牌市占率**：按日期 × 品牌的市占率可保留全部月份或关键月份，不省略品牌。
- **品牌客单价**：10 个品牌客单价完整列出。
- **品牌价格段分布**：每个品牌下各价格段、各日期的销量可做成紧凑表（品牌、价格段、日期、销量），或按品牌汇总后保留各价格段占比，**不省略任何品牌、任何价格段**。

**长度与模型选择**（见 3.4）。

### 3.3 后端如何用这份上下文

- 后端收到 `{ question, context }` 后：
  - **System prompt** 固定为（中文）：你是本报表的 AI 助手。**必须仅根据下面提供的【数据上下文】用中文回答问题**；若上下文没有相关信息则明确说「当前数据中无法得出」；不要编造数字。**不要回答与页面本身相关的内容**：不介绍或解释页面布局、界面设计、页面上已有的图表/报表有哪些或如何使用；只针对「大盘」与「分品牌」两份数据做针对性解读与回答。
  - 将前端传来的 `context` 原样拼进 system prompt（例如：`【数据上下文】\n{context}`）。
  - **User message** 即为用户本次 `question`。
  - 调用国内模型得到回复后，将回复返回前端（可流式或一次性 JSON）。

### 3.4 上下文长度与「完整」的取舍

在保证**不省略任何品牌、任何维度**的前提下，有两种可行策略：

**策略一（推荐，常规模型）**：**完整覆盖的结构化摘要**

- 前端 `buildDataContext()` 按 3.2 的「紧凑表格/列表」规则，把大盘与分品牌**全部维度、全部品牌、全部关键指标**都放进 context，不删品牌、不删渠道/价格段，仅做格式压缩（如 JSON 转成「品牌\t销量\t销售额\t占比」一行一行）。
- 目标长度：**2 万～4 万字符**（约 8k～16k tokens），这样 32k 上下文的模型（通义、智谱等常规型号）即可跑通，单次成本与延迟可控。
- 若实际构建后仍超长：优先保证**分品牌完整**（10 个品牌 × 各块数据全保留），大盘侧可对「按日期」类维度只保留最近 24～36 个月并在 context 开头注明「大盘时间范围已截断至最近 N 个月」。

**策略二（长上下文模型）**：**更大 context，保留更多明细**

- 若希望模型看到更多原始明细（如更多月份、更多小数位），可选用**长上下文模型**（如 通义 qwen-long、月之暗面 Moonshot、DeepSeek 128k），将 context 上限提高到 **6 万～10 万字符**（约 2 万～3 万 tokens），前端仍做「完整覆盖」、不省略品牌与维度，但可少做时间截断、保留更多日期与价格段明细。
- 代价：单次 token 消耗与延迟上升，需在 Worker 中配置对应模型名与 API。

**建议**：先按策略一实现并联调，确认模型能基于「完整覆盖」的 2 万～4 万字符 context 正确回答；若业务上确实需要更多明细再切到策略二并选用长上下文模型。

## 4. 国内模型配置方式（Worker 内用 Secret）

### 4.1 通义千问（阿里云 百炼 / DashScope，推荐）

**在百炼平台的操作步骤：**

1. **登录**：打开 [阿里云百炼](https://bailian.console.aliyun.com/)（或 [灵积 DashScope](https://dashscope.console.aliyun.com/)），使用阿里云账号登录；若无账号需先注册阿里云。
2. **开通服务**：在控制台中找到「模型服务」或「通义千问」，若未开通则按提示开通（部分模型按量计费，需账户有一定余额或开通后付费）。
3. **创建 API Key**：
   - 百炼：在控制台侧栏或「API-KEY 管理」「密钥管理」等入口进入，点击「创建 API Key」或「创建密钥」。
   - 灵积/DashScope：进入 [API-KEY 管理](https://dashscope.console.aliyun.com/apiKey)，点击「创建新的 API-KEY」。
   - 创建后**立即复制并保存** Key（通常只显示一次），后续在 Cloudflare Worker 的 Secret 中使用。
4. **确认模型与接口**：通义千问的开放 API 一般通过 **DashScope** 对外提供，模型名如 `qwen-turbo`、`qwen-plus`、`qwen-max`；接口为兼容 OpenAI 格式：`https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions`（若百炼控制台文档中给出其他 endpoint，以控制台为准）。
5. **在 Worker 中配置**：在 Cloudflare Worker 项目目录执行 `wrangler secret put DASHSCOPE_API_KEY`，粘贴上述 API Key；请求时 Header 为 `Authorization: Bearer <你的Key>`，Body 为 `{ "model": "qwen-turbo", "messages": [ {"role": "system", "content": "..."}, {"role": "user", "content": "..."} ] }`。

若控制台名称或菜单位置有更新，以阿里云当前页面的「API Key」「密钥管理」「模型服务」等入口为准；也可在帮助中心搜索「通义千问 API」或「百炼 调用」查看最新文档。

### 4.2 智谱 ChatGLM

- **申请**： [智谱开放平台](https://open.bigmodel.cn/) 注册并创建 API Key。
- **接口**：OpenAI 兼容，如 `https://open.bigmodel.cn/api/paas/v4/chat/completions`，模型 `glm-4-flash`、`glm-4`。
- **Worker 配置**：`wrangler secret put ZHIPU_API_KEY`，Header `Authorization: Bearer <key>`，Body 同上。

### 4.3 其他（文心、月之暗面、DeepSeek 等）

- 文心： [百度千帆](https://cloud.baidu.com/product/wenxinworkshop) 创建应用，按文档鉴权。
- 月之暗面： [开放平台](https://platform.moonshot.cn/) 获取 Key，OpenAI 兼容，适合长上下文。
- DeepSeek： [开放平台](https://platform.deepseek.com/) 获取 Key，中文与性价比都不错。

**统一原则**：先实现一家（如通义或智谱），后续切换厂商只需改 Worker 内请求 URL 与 body，前端不变。

## 5. 如何接入到网页中

### 5.1 前端改动（均在现有 index.html / charts.html 内）

**UI**

- **默认状态**：页面右下角仅显示一个**悬浮按钮**（如「AI 助手」图标或文案），不占用主内容区域。
- **展开状态**：用户点击悬浮按钮后，在按钮附近（如其上方或侧方）展开一个**悬浮的聊天框**（可固定宽高、圆角、阴影），内含：简短说明（本助手仅根据当前报表数据回答）、**数据范围选择**、**对话区域**、**输入框 + 发送按钮**、加载状态与错误提示。再次点击按钮或聊天框外区域可收起聊天框，恢复为仅显示悬浮按钮。
  - **数据范围选择**：在输入框上方（或旁边）提供选项，让用户选择本次提问围绕的数据范围：**仅大盘数据** / **仅品牌数据** / **两者**（默认）。单选框或下拉均可；若用户不选则视为「两者」。
  - **聊天记录不长期存储**：仅保存在当前页面的内存中，刷新页面即清空，每次打开/刷新都是一次新的对话，无需后端存储或本地持久化。

**数据上下文函数**

- 新增 `buildDataContext(scope)`：参数 `scope` 为 `'dapan'` | `'brand'` | `'both'`，由当前用户选择决定。内部按 scope 决定拼入哪些数据：
  - **仅大盘**（`'dapan'`）：只调用 `getDaPan()`，按 3.2 大盘侧规则生成紧凑字符串，不包含分品牌数据。
  - **仅品牌**（`'brand'`）：只调用 `getFenPinPai()`，按 3.2 分品牌侧规则生成紧凑字符串，不包含大盘数据。
  - **两者**（`'both'` 或未选）：同时调用 `getDaPan()`、`getFenPinPai()`，按 3.2 完整覆盖、紧凑格式拼成一段字符串（先大盘后分品牌）。
- 不论 scope 为何，对应数据侧均**不省略任何品牌、任何维度**，仅做格式压缩；总长目标仍为 2 万～4 万字符（策略一），单选一侧时 context 更短、模型负担更小。

**请求逻辑**

- 用户点击发送时：读取输入框内容为 `question`，读取**数据范围选择**得到 `scope`（若未选则 `'both'`）；若未加载对应数据（如选了仅品牌但分品牌未加载）可提示「请先加载对应数据」。否则调用 `buildDataContext(scope)` 得到 `context`，`fetch` 到 Cloudflare Worker 的 URL，方法 POST，body 为 `JSON.stringify({ question, context })`。
- 处理响应：若为 JSON 则取 `reply` 展示；若为流式则逐 chunk 追加到回复区域。

**跨域**

- Worker 需设置 CORS：允许报表页所在域名（如 `https://qbt-datavisualization.pages.dev`）的 `Origin`，对 OPTIONS 返回 204。

### 5.2 后端：Cloudflare Worker（已采用）

- 单独建一个 **Cloudflare Worker** 项目，接收 `POST` 请求体中的 `question`、`context`，组装中文 system prompt 后通过 `fetch` 请求国内大模型 API，从响应中取出回复并返回 JSON `{ reply: "..." }`。API Key 放在 Worker 的 **Secret** 中，前端不接触。
- **后续修改**：凡后端逻辑或配置变更（改 prompt、换模型、修接口），只需改 Worker 代码并执行 **`wrangler deploy`** 即可，无需维护其它服务器或国内云函数。
- **实现要点**：路由 `POST /chat` 或 `POST /`，校验 body 含 `question`、`context`；将 system 设为「固定中文说明 + context」，user 设为 question；按所选厂商文档发 HTTP 请求；统一返回 JSON `{ reply: "..." }`。

## 6. 实施步骤建议

1. **后端**：选定一家国内模型（建议先通义或智谱），新建 **Cloudflare Worker** 项目，实现 `POST` 接收 `question` + `context`，组装中文 system prompt 并调用该厂商 API，返回 JSON `{ reply: "..." }`；配置 CORS；API Key 设为 Worker Secret。后续修改仅需更新 Worker 并重新部署（`wrangler deploy`）。
2. **前端**：在 index.html 中增加默认仅显示的**悬浮按钮**，点击后展开**悬浮聊天框**；在聊天框内增加**数据范围选择**（仅大盘 / 仅品牌 / 两者，默认两者）；实现 `buildDataContext(scope)` 以及调用 Worker URL 的 `fetch` 与展示逻辑；未加载对应数据时提示「请先加载对应数据」；界面与提示均为中文。对话消息仅保存在当前页面的内存中，页面刷新后自然清空，即每次都是新聊天框。
3. **联调**：用报表页对 Worker URL 发起请求，确认回答内容基于传入的 context 且为中文。
4. **同步 charts.html**：将相同 UI 与调用逻辑复制到 charts.html（或抽成公共脚本再引入）。
5. **可选**：增加流式输出；或通过环境变量切换为其它国内厂商。

## 7. 部署后你要做的

按当前实现，你只需完成以下 3 步即可让 AI 助手生效：

1. **配置 API Key**：在项目根目录下进入 `worker` 目录，执行 `wrangler secret put DASHSCOPE_API_KEY`，按提示粘贴你在阿里云百炼/灵积获取的通义千问 API Key。
2. **部署 Worker**：在同一目录执行 `wrangler deploy`，部署成功后终端会输出该 Worker 的访问地址（形如 `https://qbt-ai-assistant.xxx.workers.dev`）。
3. **填写前端地址**：将上一步得到的 Worker 地址填入前端的 **`AI_ASSISTANT_WORKER_URL`**。该变量在 **index.html** 与 **charts.html** 的脚本开头各有一处（注释为「部署 Worker 后在此填写」），两处都改为同一 Worker 地址即可。

完成后刷新报表页，登录后即可使用右下角 AI 助手按钮进行提问。

## 8. 风险与注意点

- **上下文长度**：采用「完整覆盖」设计（见 3.2、3.4），目标 2 万～4 万字符；若超长则优先保证分品牌完整，大盘时间维度可截断并注明。避免无意义冗长或超出所选模型 context 上限。
- **密钥安全**：API Key 只存在于 Worker Secret，前端永远不接触。
- **合规与用量**：接入国内模型需遵守各厂使用条款与计费规则；建议对接口做简单限流或仅内网/授权使用。
- **无数据情况**：前端在未加载大盘/分品牌时，应禁止发送或提示先加载数据，避免传空 context 导致模型乱答。
- **国内访问**：已采用 Cloudflare Worker 时，国内用户请求会经 Worker 再访问国内 API；若延迟或厂商境外策略不满足需求，可后续再迁至国内云函数。

按上述方案即可在现有报表页中增加「基于两份数据源、面向国内中文用户」的 AI 助手，且后续后端迭代只需更新 Cloudflare Worker 部署即可。
