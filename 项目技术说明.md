# QBT数据可视化平台 - 技术说明文档

## 一、项目概述

**QBT数据可视化平台**是一个基于浏览器的行业数据分析工具，主要用于展示学习机/教育硬件行业的销售数据、品牌趋势、市场占有率等关键指标。

**核心功能：**
- 📊 行业大盘数据展示（销量、销售额、渠道分布）
- 📈 品牌销量/销额趋势分析
- 🏆 品牌市场占有率排行
- 📱 多渠道数据整合（天猫、京东、抖音等）
- 📅 日销、月销、同比等多维度数据对比

**使用场景：**
企业内部数据分析、管理层决策支持、运营团队日常数据监控

## 三、前端技术（用户看到的东西）

### 3.1 什么是前端？
前端就是用户在浏览器中看到的所有界面和交互效果。当你打开网站，看到的按钮、图表、下拉菜单等都是前端。

### 3.2 我们用了什么技术？

| 技术 | 通俗解释 | 作用 |
|------|---------|------|
| **HTML5** | 网页的骨架 | 定义页面的结构，比如哪里放标题、哪里放图表 |
| **JavaScript** | 网页的大脑 | 处理用户的点击、计算数据、和服务器通信 |
| **CSS** | 网页的衣服 | 让页面好看，设置颜色、字体、布局 |
| **ECharts** | 专业的画图工具 | 把枯燥的数字变成漂亮的折线图、柱状图、饼图 |
| **FontAwesome** | 图标库 | 提供各种小图标，如折线图图标、柱状图图标 |

### 3.3 前端特点
- **响应式设计**：自动适应电脑、平板、手机不同屏幕大小
- **无需安装**：用户只需要浏览器就能使用
- **纯静态页面**：没有复杂的服务器渲染，加载速度快

## 四、后端技术（背后的数据处理）

### 4.1 什么是后端？
后端就是服务器端的程序，处理用户看不到的逻辑：验证密码是否正确、从数据库读取数据、计算复杂的统计指标等。

### 4.2 Serverless架构（无服务器架构）

**传统架构 vs Serverless：**

| 传统架构 | Serverless（本项目用的） |
|---------|------------------------|
| 需要租一台24小时运行的服务器 | 不需要租服务器，代码在需要时才运行 |
| 服务器空闲时也要付费 | 只有代码运行时收费，空闲时不花钱 |
| 需要自己维护服务器系统 | 服务商（Cloudflare）自动维护 |
| 扩展麻烦，需要手动加服务器 | 自动扩展，瞬间应对高并发 |

**通俗理解：**
Serverless就像"按需使用的出租车"，你需要时它出现，用完就消失；传统服务器就像"自己买辆车停在那里"，不管用不用都要养着。

## 六、账号体系与安全机制

### 6.1 登录流程（JWT认证）

```
用户输入账号密码
       │
       ▼
前端发送登录请求
       │
       ▼
后端验证密码（对比哈希值）
       │
       ▼
生成JWT令牌（包含用户ID和签名）
       │
       ▼
返回Token给前端
       │
       ▼
前端存储Token（Cookie + LocalStorage双保险）
       │
       ▼
后续请求携带Token，后端验证Token有效性
```

### 6.2 什么是JWT？
JWT（JSON Web Token）就像一张"数字通行证"：
- 🎫 **包含信息**：用户ID、过期时间等
- 🔒 **无法伪造**：带有数字签名，篡改后失效
- ⏰ **有时效性**：过期后自动失效，需要重新登录

### 6.3 安全防护措施

| 安全措施 | 说明 | 防护对象 |
|---------|------|---------|
| **密码哈希** | 使用PBKDF2算法加密，即使数据库泄露也无法还原密码 | 数据库泄露 |
| **JWT签名** | 使用密钥签名Token，防止伪造 | Token篡改 |
| **登录失败限制** | 同一IP连续失败5次封禁15分钟 | 暴力破解攻击 |
| **CORS跨域限制** | 只允许特定域名访问API | CSRF攻击 |
| **XSS防护** | 输出内容转义，防止恶意脚本注入 | XSS攻击 |
| **HTTPS传输** | 全程SSL加密，防止数据被窃听 | 中间人攻击 |

### 7.1 为什么要区分？

想象你在装修房子：
- **测试服** = 样板间（可以随便试，弄坏了也没关系）
- **正式服** = 实际居住的房子（必须稳定，不能出问题）

### 7.2 两套环境对比

| 维度 | 正式服（生产环境） | 测试服（开发环境） |
|------|------------------|------------------|
| **域名** | `qbt-datavisualization.pages.dev` | `qbt-datavisualization-test.pages.dev` |
| **用途** | 供日常使用的稳定版本 | 新功能测试验证 |
| **数据** | 真实的行业数据 | 同正式服（只读，不修改） |
| **账号** | 正式用户账号 | 独立的测试账号（test / xbs2026test） |
| **数据库** | `qbt-auth` | `qbt-auth-test`（完全隔离） |
| **更新频率** | 稳定版本，谨慎更新 | 频繁更新，随时回滚 |

## 十、缓存策略

### 10.1 为什么需要缓存？

**问题：**
- 每次用户访问都从飞书读取数据 → 飞书API调用次数有限制
- 飞书API有速率限制，频繁调用会被封禁
- 数据读取慢，用户体验差

**解决方案：**
数据缓存（把数据暂存到D1数据库，定期更新）

### 10.2 缓存机制

```
用户请求数据
     │
     ▼
检查缓存是否存在？
     │
     ├─ 是 ──▶ 检查是否过期？
     │              │
     │              ├─ 未过期 ──▶ 返回缓存数据（快）
     │              └─ 已过期 ──▶ 从飞书读取新数据
     │                                  │
     └─ 否 ────────────────────────────┘
                                        ▼
                                  更新缓存
                                        │
                                        ▼
                                  返回数据
```

**缓存时间：**
- 行业大盘数据：48小时
- 品牌数据：48小时
- 日销数据：2小时（更新频繁）

## 十一、技术选型理由

### 11.1 为什么选择Cloudflare生态？

| 因素 | Cloudflare方案 | 传统方案（如阿里云/腾讯云） |
|------|---------------|------------------------|
| **成本** | 免费额度充足 | 最低也要几十元/月 |
| **维护** | 全自动，无需运维 | 需要专业运维人员 |
| **全球访问** | 自动CDN加速 | 需额外购买CDN服务 |
| **安全性** | 内置DDoS防护、WAF | 需额外购买安全服务 |
| **扩展性** | 自动扩展，无感知 | 需要手动扩容 |
| **学习成本** | 较低，文档完善 | 较高，需要掌握更多概念 |

### 11.2 为什么不使用React/Vue等框架？

**项目特点决定：**
- 纯数据展示，交互简单（主要是图表）
- 使用原生JavaScript足够，无需复杂框架
- 减少依赖，降低维护成本
- 加载速度更快（没有框架代码）

**适用场景对比：**
- ✅ 本项目：数据大屏、图表展示 → 原生JS足够
- ❌ 不适用：复杂后台管理系统、大量表单交互 → 需要React/Vue

|---------|---------|---------|
| 页面空白 | JavaScript报错 | 按F12打开控制台查看错误 |
